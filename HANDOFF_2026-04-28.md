### #106 — IcsCodec TZID

`parseIcsInstant` only handled UTC `Z` form, plus the field loop dropped everything after `;` in the property name. So `DTSTART;TZID=America/Los_Angeles:20260427T140000` had its TZID stripped, hit the `else` branch, returned null, and the whole VEVENT was silently dropped.

Fix: a `FieldEntry(value, params)` data class captures both halves of a content line, and `parseIcsInstant(raw, tzid)` now accepts UTC, TZID-anchored, floating local, and date-only forms. Unparseable TZID falls back to system zone.

### #105 — pin parse asymmetry

`MemoRepository.readPinnedFromFrontMatter` tolerated quoted values (`pinned: "true"`) while `FrontMatterCodec.applyPin/strip` rejected the same input as user YAML. Function was internal-only and used only by the test. Deleted it; the test now goes through `FrontMatterCodec.parse`. Added a regression for quoted-value treatment.

### #113 — credential-change push retry

`PushWorker` returns `Result.failure()` on 401, leaving dirty rows parked. `SettingsStore.update` now detects PAT/owner/repo change + `isConfigured` and calls a new `SyncScheduler.enqueuePushAfterCredentialChange` (uses `ExistingWorkPolicy.REPLACE` to abandon any in-flight worker on stale credentials). Wrapped in `runCatching` because unit tests don't initialise WorkManager.

### #140 — inline Settings validation

PAT regex (GitHub's published prefix taxonomy), provider URL https-only + URI parse, model whitespace, key non-blank. Errors live on UiState (`patError`, `aiProviderUrlError`, `aiModelError`, `aiApiKeyError`); `SettingsScreen` threads them into `OutlinedTextField.isError` + `supportingText`. Validators are on the `companion object` so `SettingsValidationTest` (16 cases) doesn't need a ViewModel + DataStore harness.

## Engineering invariants to preserve

- **Sandbox edits are not parallel-safe**: don't dispatch >1 agent that writes the same file in the same wave. We had multiple wipes earlier.
- **PushWorker reads PAT from `SettingsStore.current()` at start of `doWork()`** — credential changes mid-flight are picked up by the next worker, not the current one. The new REPLACE-policy retry handles this.
- **`WidgetRefresher` uses `MutableSharedFlow + debounce(400ms) + DROP_OLDEST`** — do not revert to `pendingJob.cancel() + delay(250)`. Glance has a 45-50s session lock and drops updates rather than queueing them.
- **`FrontMatterCodec.applyPin/strip` is the only place that reads/writes the pin marker** — `MemoRepository.readPinnedFromFrontMatter` is gone. Don't reintroduce a parallel parser.
- **All Settings persistence goes through `SettingsStore.update` or `switchAccount`** — both now have credential-change side effects (widget refresh + push retry). Don't bypass with a direct `settingsDataStore.edit`.
- **Validators are on the companion object** — keep them pure, no ViewModel state access. The screen calls `vm.save()` / `vm.saveAiConfig()`, which run validators before any IO.
